## Принцип

Снизу вверх. Один этап = один связный проверяемый кусок. Не переходи дальше, пока DoD не выполнен и тесты зелёные. Меньше кода — меньше ошибок: у каждой строки есть текущий потребитель.

### G0 — Скелет: два крейта, границы, CI

**Выход.** Cargo-воркспейс: `logic/` и `canister/`. `config/testnet.toml`, `config/mainnet.toml` по схеме game-spec §7. `dfx.json`. CI: fmt, clippy, test, и пять структурных линтов ниже.

**Структурные линты (это и есть суть этапа).**
1. `cargo tree -p conditional-tasks-logic --edges normal` печатает **только сам крейт**. Ноль зависимостей.
2. `grep -rE 'ic_cdk|std::(fs|net|time)|reqwest' logic/src/` → пусто.
3. `grep -riE 'solana|ed25519|sha256|candid' logic/src/` → пусто. Логика не знает про чейн и криптографию.
4. `grep -riE 'transfer|approve|splitter|fee|bps|treasury' canister/src/` → пусто; `grep -r 'http_request' canister/src/` → пусто. Канистра не двигает деньги и не читает внешние сети.
5. Парсер `.did`: update-методы — ровно `{register_task, accept, decline, done, vote, set_channel_params}`. Любой сверх списка роняет CI.

**DoD.** Дерево совпадает с картой в `CLAUDE.md`. Пять линтов зелёные на пустом репозитории. Ни одного сетевого значения, адреса или principal в коде.

### G1 — `logic` — чистая машина состояний

**Вход.** game-spec §3, §6.

**Выход.** Крейт `conditional-tasks-logic`. В нём: `Task`, `State`, `Action`, `Vote`, `Verdict`, функция перехода `step(task, action, now) -> Result<...>`, подсчёт `verdict(votes)`, константы правил (`MIN_DURATION`, `MAX_DURATION`, `DEADLINE_MARGIN`, `MIN_VOTE_WEIGHT`), `LOGIC_VERSION`. Окно голосования — параметр рождения задания (профиль конфига), не константа. Время — аргумент. Адреса и ключи — непрозрачные байты.

Заголовок `lib.rs`:
```rust
#![forbid(unsafe_code)]
#![deny(clippy::unwrap_used, clippy::expect_used, clippy::panic,
        clippy::arithmetic_side_effects, clippy::indexing_slicing)]
```

**DoD.** Property-тесты (`proptest`, dev-dependency) доказывают:
- **недостижимость** — переход, отсутствующий на диаграмме game-spec §3, возвращает ошибку при любом входе; матч по `(State, Action)` исчерпывающий, без `_ =>`;
- **единственность вердикта** — после `Verdict` ни одна последовательность действий не меняет ни состояние, ни вердикт;
- **большинство** — `settle` ⇔ Σвес «выполнено» > Σвес «не выполнено»; ничья и пустое голосование дают `cancel`; фаззом по наборам голосов;
- **дедуп** — повторный голос адреса отвергается и не меняет счёт;
- **время** — «принять» и «готово» после истечения `duration` невозможны; истечение необратимо;
- **детерминизм** — одна последовательность `(action, now)` даёт побитово одно состояние.

Плюс: `LOGIC_VERSION` определён и покрыт тестом на неизменность при пересборке.

### G2 — Канистра: задания и подписи участников

**Вход.** game-spec §3, §4, §5, §7.

**Выход.** Канистра поверх `logic`. Update-методы: `register_task`, `accept`, `decline`, `done`, `set_channel_params`. Проверка Ed25519-подписей кошельков над сообщением game-spec §4. `task_id` выводится через `crown-derive` из полей рождения и конфига. Таймеры: истечение `duration` переводит задание в `cancel`-вердикт без чьего-либо вызова. Состояние — в стабильной памяти; корень состояния — в `set_certified_data`, `query` отдаёт `data_certificate()`.

**DoD.**
- `task_id` совпадает с адресом реального эскроу: соль запиннена кросс-инструментальным вектором (python3 hashlib), арифметика адреса — `crown-derive` с его собственным паритетом; канистра и клиент выводят один `task_id`.
- Регистрация отвергается при: `gross < max(min_gross, флор)`, `duration ∉ [MIN_DURATION, MAX_DURATION]`, `deadline < регистрация + duration + VOTING_PERIOD + DEADLINE_MARGIN`, `enabled = false`, повторный `task_id`.
- Подписи: чужой ключ, чужое задание, чужая канистра, чужое действие — отвергнуты. Повтор `set_channel_params` со старым счётчиком — отвергнут. `accept`/`decline`/`done` идемпотентны в пределах задания.
- «Принять» и «отказаться» работают из состояний по диаграмме и только из них; «принять» после `duration` — отказ. Таймерные переходы проверены сдвигом времени (PocketIC).
- Сертификат состояния проверяется офчейн против root key — есть тест.
- Линт `.did` зелёный; `vote` ещё не существует.

### G3 — Голосование весом книги

**Вход.** game-spec §6.

**Выход.** Update-метод `vote`. Порядок: подпись кошелька → дедуп `(task_id, address)` → межканистерный вызов `crown-index.get_reputation(chain, voter, streamer)` → запись `(адрес, выбор, вес)`. Регистрация получает `donor_reputation` из той же книги — заглушка «ноль» из G2 умирает здесь. По таймеру конца `VOTING_PERIOD` канистра считает вердикт функцией из `logic` и пишет его в стабильную память. Состав голосования — в certified query.

**DoD.**
- Голос с весом ниже `MIN_VOTE_WEIGHT` отвергнут; вес в записи равен значению книги в момент обработки.
- Голос вне `VOTING`, повторный голос — отвергнуты.
- Ошибка вызова `crown-index` возвращает ошибку голосующему и **не** записывает голос; повтор после восстановления проходит.
- Вердикт канистры совпадает с `logic::verdict` на одних векторах, включая ничью и пустое голосование → `cancel`.
- Вердикт записан ровно один раз; повторный тик таймера ничего не меняет.
- В PocketIC книга — мок за init-оверрайдом (на mainnet-профиле оверрайды трапаются); вес из **реальной** книги доказывается сквозным прогоном G4, где devnet-донат порождает репутацию, которой голосуют.

### G4 — Вердикт: threshold-подпись + e2e через реальную сеть

**Вход.** game-spec §8.

**Выход.** Модуль подписи: tSchnorr (Ed25519) для message формы, побайтово по game-spec §8. Подпись запрашивается таймером после записи вердикта, хранится рядом с ним, отдаётся через `query`. Ретрай при отказе management canister — тот же исход, идемпотентно. E2e-скрипт в `scripts/` против devnet.

**DoD.**
- Локально: подпись верифицируется Ed25519 против pubkey канистры и ни для какого другого `(эскроу, исход)`.
- **Devnet:** `create_escrow` с `resolver` = ключ канистры → полный цикл в канистре → вердикт `settle` → `claim(0, sig)` реальным эскроу проходит, подпись едет инструкцией `ed25519_program` → сплиттер ядра эмитит `Settled(payer = escrow)` → локальный `crown-index` начисляет `+gross` **донору**.
- `claim(1, sig)` и `refund()` не эмитят `Settled`; книга не меняется.
- Запрос подписи для другого исхода после записанного вердикта — отказ; негативный тест.
- Подпись невалидна для чужого эскроу — негативный тест против реального контракта.
- **`git diff` в `crown-core` и `crown-factory` пуст.** Это и есть настоящий тест шва.

После зелёного G4 игра готова в devnet-объёме: полный цикл «регистрация → эскроу → принятие → голосование → вердикт → книга» работает без сервера.
