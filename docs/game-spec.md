# Условные задания — Спецификация

**v1.0 · Канистра-резолвер · Больше ничего**

**Класс B.** Холд и начисление репутации донору. Первая игра на форме `two-outcome` из `crown-factory`.

Контрактов в этой игре **ноль** (factory-spec §7): форма задеплоена, её фабрики пиннуты в `factories[]` конфига ядра, игра лишь передаёт свой `resolver` в `createEscrow`. Ядро не изменяется. Фабрика не изменяется. Игра читает книгу и подписывает вердикт; деньги двигает эскроу.

Читается вместе с `crown-core/docs/core-spec.md` и `crown-factory/docs/factory-spec.md`. Термины оттуда не переопределяются.

---

## 1. Что происходит

Донор кладёт USDC в эскроу и прикладывает текст задания. С этого момента идёт срок выполнения, заданный донором. Стример видит текст приватно и решает, брать ли. Взял — текст раскрывается публично. Заявил «готово» — окно голосования (час в проде), вес голоса равен репутации у этого стримера. Канистра считает голоса и подписывает исход.

Деньги уходят стримеру через сплиттер (репутация начисляется донору) либо возвращаются донору (репутации нет).

Заработать больше внесённого нельзя ни одной стороне.

## 2. Участники

| | Что это | Держит деньги | Может врать |
|---|---|---|---|
| Донор | человек | нет | — |
| Стример | человек | нет | — |
| Голосующие | адреса с репутацией у стримера на этой сети | нет | — |
| **Игровая канистра** | канистра ICP, threshold-ключи, blackholed в проде | нет | нет* |
| **Эскроу** | контракт формы `two-outcome`, immutable | да, до расчёта | нет |
| **Сплиттер** | ядро | нет | нет |
| **crown-index** | ядро, книга | нет | нет |
| Сервер | `crown-app` | нет | **да** |
| Модератор канала | человек, на сервере | нет | **да** |

\* Для фабрики резолвер — недоверенный (factory-spec §8): границы его вранья — `settle` или `cancel`, больше он не может ничего. Игра сужает и это: код зафиксирован блэкхолом, приватного ключа не существует — подпись собирает подсеть ICP.

Сервер и модератор находятся вне денежного пути и вне пути вердикта.

Заданий на канале одновременно сколько угодно. Каждое — свой эскроу, свой таймер, своё голосование. Между собой они не связаны.

## 3. Машина состояний

Часы игры — канистра. Один таймер: `duration`, стартует при регистрации задания.

```
    регистрация (ingress, подпись донора) + createEscrow (транзакция донора)
                          │  старт duration
                          ▼
                    ┌───────────┐
                    │  CREATED  │  текст виден только стримеру
                    └─────┬─────┘
        принять ──────────┼────────── отказаться
               │                            │
               ▼                            │
        ┌───────────┐                       │
        │ ACCEPTED  │  текст раскрыт        │
        └─────┬─────┘                       │
   готово ────┼──── истёк duration ─────────┤
          │         или отказался           │
          ▼                                 │
    ┌───────────┐                           │
    │  VOTING   │ голосование, вес=репутация│
    └─────┬─────┘                           │
          │ канистра подсчитала             │
          ▼                                 ▼
    ┌───────────┐                    ┌───────────┐
    │ VERDICT:  │                    │ VERDICT:  │
    │  settle   │                    │  cancel   │
    └─────┬─────┘                    └─────┬─────┘
          │  claim(0, sig)                 │  claim(1, sig)
          ▼                                ▼
эскроу → СПЛИТТЕР → стример 97%      эскроу → донор 100%
                  → казна 3%         Settled нет, репутации нет
Settled(payer = эскроу)
индексатор читает donor() → репутация донору

────────────────────────────────────────────────────────────
REFUND: кто угодно, строго после DEADLINE, эскроу → донор.
        Страховка от мёртвой канистры. Репутации нет.
```

Принять можно в любой момент до истечения `duration`, вплоть до последней секунды.

Отказаться можно из `CREATED` и из `ACCEPTED`, до «готово». Отказ бесплатен для стримера: это подпись, не транзакция; `claim(1)` оплачивает кто угодно (§12). Отказ после принятия освобождает деньги донора сразу, не дожидаясь конца `duration`.

Истёк `duration` без принятия → `cancel`. Истёк `duration` без «готово» → `cancel`. Переходы по времени делает таймер канистры.

Вердикт фиксируется в стабильной памяти **до** запроса подписи и не переписывается. Канистра никогда не подписывает два разных исхода одного эскроу; повторная подпись того же исхода (ретрай threshold-подписи) допустима и ничего не меняет.

## 4. Кнопки: транзакция или подпись

| Кнопка | Тип | Кто платит | Куда идёт |
|---|---|---|---|
| Зарегистрировать задание | **подпись** донора | никто | канистра: поля рождения + `sha256(text ‖ text_salt)` |
| Профинансировать | **транзакция** | донор | EVM: `approve` + `createEscrow`; Solana: `create_escrow` |
| Показать текст | — | — | UI |
| Принять | **подпись** | никто | канистра |
| Отказаться | **подпись** | никто | канистра → вердикт `cancel` |
| Готово | **подпись** | никто | канистра → старт голосования |
| Голос | **подпись** | никто | канистра, напрямую, минуя сервер |
| Забрать деньги | **транзакция** | кто нажал | эскроу `claim(outcome, sig)` |

Ончейн-транзакций за жизненный цикл: две на Solana (`create_escrow`, `claim`), три на EVM (`approve`, `createEscrow`, `claim`; permit-вариант фабрики сознательно отложен, `post-factory.md`). Плюс `refund()` в аварийном случае.

Канистра не отправляет транзакций и не держит нативного токена ни на одной сети. Она подписывает вердикт threshold-ключом (ECDSA secp256k1 на EVM, Ed25519 на Solana) и отдаёт подпись через `query`.

`claim` зовут кто угодно. Получатели зафиксированы при рождении эскроу.

**`task_id` ≡ адрес эскроу.** Канистра выводит его при регистрации сама, той же арифметикой, что и индексатор ядра: EVM — `create2(factory, salt, ESCROW_INIT_CODE_HASH)`, Solana — PDA `[b"escrow", salt]` фабрики; `salt` считается из полей рождения. Арифметика — внешний крейт `crown-derive`.

Все подписи участников — кошельком той сети, где живёт эскроу, и привязаны к `(canister_id, task_id, действие)`. Сообщение: `"crown:conditional-tasks" ‖ canister_id ‖ task_id ‖ действие ‖ параметры действия` (для голоса — выбор; для параметров канала — значения и монотонный счётчик, иначе старая подпись проигрывается заново). EVM-кошельки подписывают его через EIP-191 `personal_sign`, Solana-кошельки — Ed25519 над теми же байтами. Подпись из одного задания в другом невалидна, из другой канистры — тоже. Подписи летят в канистру напрямую и не проходят через сервер.

**Канистра не читает внешние сети.** Она знает об эскроу то, что заявлено при регистрации, и этого достаточно: вердикт для несуществующего или недофинансированного эскроу бесполезен — деньги двигает только сам эскроу. Профинансирован ли адрес — проверяет стример (и показывает сервер) до «принять».

## 5. Хранение и раскрытие текста

| Что | Где | Кто может подделать |
|---|---|---|
| поля рождения: `DONOR, STREAMER, GROSS, DEADLINE, RESOLVER, NONCE` | нотаризованы адресом эскроу через `salt`; Solana — читаются из аккаунта эскроу, EVM — публичны только `donor()` и `salt()` | никто |
| `sha256(text ‖ text_salt)` | канистра | никто |
| `text`, `text_salt` | сервер | оператор — только скрыть |
| подписи: регистрация / принять / готово / отказаться | канистра | никто |
| голоса: адрес, выбор, вес, подпись | канистра | никто |
| вердикт + подпись резолвера | канистра, `query` | никто |
| `min_gross`, `min_reputation`, `enabled` | канистра | никто |
| тексты, аватарки, лидерборды, баны | сервер | оператор |

Хеш попадает в канистру при регистрации, до того как стример нажмёт «принять». Сервер отдаёт стримеру `text + text_salt`; стример сверяет хеш локально. `text_salt` — случайные 32 байта донора, защищают короткий текст от брутфорса по хешу. Не путать с `salt` эскроу — солью деривации адреса.

**Раскрытие.**

```
1. донор → канистра     регистрация: поля рождения, sha256(text ‖ text_salt), подпись
2. донор → фабрика      createEscrow — эскроу существует и профинансирован
3. донор → сервер       text, text_salt
4. сервер → стример     text, text_salt      пока состояние CREATED
5. стример → канистра   подпись «accept»     ingress, напрямую, минуя сервер
6. канистра             состояние := ACCEPTED, сертифицировано
7. сервер поллит канистру, видит ACCEPTED → публикует text, text_salt
8. кто угодно           sha256(text ‖ text_salt) == hash из канистры
```

Сервер не принимает подпись и не решает, состоялось ли принятие. Он читает состояние канистры и проверяет `data_certificate()` против root key NNS. Подпись, поставленная вне платформы и отправленная в канистру напрямую, раскрывает текст точно так же.

Раскрытие необратимо и не монополизировано: `text + text_salt` на шаге 4 уже у стримера, а на шаге 3 — у донора. Любой из них публикует текст где угодно, и подлинность проверяется хешем из канистры. Сервер может перестать показывать текст (§9), но не может помешать его раскрытию и не может отрицать факт принятия.

## 6. Голосование

- **Право голоса:** `book[(chain, voter, streamer)] ≥ MIN_VOTE_WEIGHT`. Порог — накопленная репутация, не платёж.
- **Вес голоса:** значение из книги в момент обработки голоса. Запрос — межканистерный вызов `get_reputation(chain, voter, streamer)` к `crown-index`; исполняется реплицированно, сертификат не нужен.
- **Выбор:** «выполнено» / «не выполнено».
- Один адрес — один голос **на задание**. Дедуп по `(task_id, address)`, проверяется до запроса веса.
- Голос — подпись кошельком, ingress-вызов в канистру. Сервер в пути голоса не стоит.
- Голосуют только носители репутации **этой сети**: книга чейн-локальна.
- Снапшота веса нет: книга не хранит времени. Вес читается на момент голоса и отстаёт на финальность плюс период ингеста.
- Порядок проверок: подпись кошелька (secp256k1 / Ed25519) → дедуп → запрос веса.

**Правило вердикта.** `settle`, если Σвес «выполнено» строго больше Σвес «не выполнено». Иначе — `cancel`, включая ничью и пустое голосование: молчание сообщества не двигает чужие деньги. Голосующие у нового канала появляются прямыми донатами через сплиттер — эмиссия репутации этой игры не требует.

Параллельные задания голосуются независимо. Один адрес голосует в каждом.

Канистра публикует состав голосования: адрес, выбор, вес. Сертифицировано, вечно.

## 7. Параметры

**Ручки стримера.** Живут в канистре, публичны. Смена — подпись стримера с монотонным счётчиком.

| | Что задаёт |
|---|---|
| `min_gross` | минимальная сумма задания, не ниже флора формы (34) |
| `min_reputation` | минимальная репутация донора для создания задания |
| `enabled` | принимает ли канал задания |

Проверяются канистрой при регистрации. Фиксируются в задании; изменение задним числом не влияет на уже созданные.

**Ручки донора.** Задаются при создании.

| | Что задаёт |
|---|---|
| `gross` | сумма |
| `duration` | время на выполнение, идёт с регистрации, публично. От `MIN_DURATION` до `MAX_DURATION` |
| `deadline` | дедлайн эскроу; канистра требует `deadline ≥ регистрация + duration + VOTING_PERIOD + DEADLINE_MARGIN` |
| `text` | текст задания |

**Константы канистры.**

| | Значение | Единица |
|---|---|---|
| `MIN_VOTE_WEIGHT` | `100_000` | minor units репутации (книга номинирована в USDC, 6 знаков) |
| `MIN_DURATION` | 1 минута | — |
| `MAX_DURATION` | 30 суток | — |
| `DEADLINE_MARGIN` | 72 часа | — |

`VOTING_PERIOD` — не константа, а значение профиля (`voting_period` в конфиге): час в проде, две минуты на тестовых сетях — иначе каждый прогон стоил бы час ожидания. Задание несёт свой период с рождения; смена профиля не трогает уже созданные.

**Флор формы.** `gross ≥ 34` minor units — проверяет **фабрика при `createEscrow`**, не сплиттер при `settle`: эскроу ниже флора не рождается. Единственное число про деньги, которое канистра знает.

**Дедлайн эскроу.** Вшит при рождении, страховка от мёртвого резолвера. После `DEADLINE` `refund()` доступен любому — в том числе донору при уже подписанном `settle`. Гонка «claim против refund» решается запасом `DEADLINE_MARGIN` и тем, что `claim` жмёт кто угодно сразу после вердикта (§12).

**Конфиг-поверхность канистры.**

```toml
crown_index   = "<principal>"                # книга, для веса голосов и min_reputation
threshold_key = "key_1"                      # имя threshold-ключа среды
voting_period = 3600                         # окно голосования; на тестнетах сжато

[[chain]]
id                    = "eth-sepolia"        # ключ книги
factory               = "0x…"                # из crown-factory deploy/
escrow_init_code_hash = "0x…"                # константа формы
evm_chain_id          = 11155111             # входит в digest вердикта
min_gross             = 34                   # флор формы, из crown-factory deploy/

[[chain]]
id        = "solana-devnet"
factory   = "<program id>"                   # program id = фабрика
domain    = "crown:two-outcome:solana-devnet"  # входит в message вердикта
min_gross = 34
```

Чего в конфиге нет: сплиттер, RPC-URL, `feeBps`, казна. Канистра не читает внешние сети и не знает про деньги ничего, кроме флора.

## 8. Контрактная часть: готовая форма

Контракты живут в `crown-factory` и уже задеплоены на обеих тестовых сетях; их инварианты доказаны там же (factory-spec §6). Игра передаёт адрес своего threshold-ключа как `resolver` — и всё.

```
                 approve(factory, gross)          (EVM; на Solana не нужен)
   ДОНОР ─────────────────────────────────► USDC
     │
     │ createEscrow(streamer, gross, deadline, resolver, nonce)
     ▼
  ФАБРИКА ── create2(salt) / PDA[b"escrow", salt] ──► ЭСКРОУ
     │          salt = hash(все поля рождения)           ▲
     └── USDC: донор ── gross ───────────────────────────┘    одна транзакция,
                                                              недофинансированного
                                                              эскроу не бывает
  КАНИСТРА ── подписывает вердикт threshold-ключом,
              кладёт подпись в query. Транзакций не шлёт.

  кто угодно ── claim(outcome, sig) ──► ЭСКРОУ
                  outcome = 0 → approve(SPLITTER, GROSS)
                                SPLITTER.donate(STREAMER, GROSS)
                                → стример 97%, казна 3%
                                → Settled(payer = эскроу)
                  outcome = 1 → USDC → DONOR. Settled нет, комиссии нет.
                  outcome ∉ {0,1} → реверт

  кто угодно ── refund() ──► ЭСКРОУ ──► DONOR     строго после DEADLINE, без подписи
```

`DONOR = msg.sender` фабрики / подписант `create_escrow` — не параметр. Конструктор эскроу на EVM без аргументов: поля рождения идут через transient storage фабрики, поэтому `ESCROW_INIT_CODE_HASH` — константа формы. На Solana `salt = sha256(donor ‖ streamer ‖ gross_le ‖ deadline_le ‖ resolver ‖ nonce_le)` считается ончейн. `nonce` задаёт донор: два задания с одинаковыми полями дали бы один адрес, второе рождение ревертнулось бы.

**Что подписывает канистра.** Единственный интерфейс игры с контрактом, побайтово:

- **EVM:** `digest = keccak256(abi.encode(block.chainid, escrow, outcome))` — сырой хеш, без EIP-191/712-префикса. Подпись — 65 байт `r ‖ s ‖ v`, `v ∈ {27, 28}`; tECDSA отдаёт `r ‖ s` и recovery id, `s` нормализуется к нижней половине кривой — `ECDSA.recover` OpenZeppelin отвергает high-s.
- **Solana:** `message = DOMAIN ‖ program_id ‖ escrow ‖ outcome`, где `DOMAIN = "crown:two-outcome:<id кластера>"`. Подпись Ed25519 (`sign_with_schnorr`). В транзакции `claim` она едет отдельной инструкцией `ed25519_program` непосредственно перед `claim`; транзакцию собирает отправитель из подписи, взятой в `query`.

`chainid` / `DOMAIN` обязательны: адреса эскроу совпадают между EVM-сетями, кластеры Solana делят program id. Вердикт не переносится ни между сетями, ни между эскроу, ни между исходами.

**RESOLVER.** Адрес threshold-ключа канистры: на EVM — eth-адрес tECDSA-ключа (secp256k1), на Solana — Ed25519-pubkey. Ключи дериватны от `canister_id`: у новой канистры они автоматически другие. Приватного ключа не существует нигде — подпись собирается подсетью ICP.

**Шов с ядром — не забота игры.** Индексатор сам читает у эскроу `salt()` и `donor()`, пересчитывает адрес от пиннутой фабрики и начисляет репутацию донору (core-spec §4). Игра пользуется той же арифметикой только чтобы вывести `task_id` при регистрации.

**Что тестирует игра.** Инварианты эскроу — двойной `claim`, чужая подпись, флор, ноль баланса — уже доказаны в `crown-factory` и здесь не повторяются. Игра тестирует своё:

- digest/message, собранные канистрой, побайтово принимаются реальным контрактом: e2e на Sepolia и devnet — `createEscrow → вердикт → claim(0)` даёт `Settled`, книга показывает `+gross` **донору**; `claim(1)` и `refund()` не дают ничего.
- Один эскроу — один исход навсегда: после записи вердикта канистра не подписывает другой ни при каком входе.
- `task_id`, выведенный канистрой, равен адресу реально рождённого эскроу — фаззом по полям рождения.
- Машина состояний: недостижимые переходы недостижимы; голос вне `VOTING` отвергается; «принять» после `duration` отвергается.
- Комиссия и репутация возникают только при `outcome = 0`.

## 9. Централизованный слой

Единственные полномочия сервера:

- **Модерация на входе.** Текст проверяется до создания задания. Не прошёл — транзакции не было.
- **Takedown.** Оператор снимает текст. Хеш остаётся, процесс продолжается без текста в UI.
- **Бан донора в канале.** Модератор канала скрывает тексты забаненного адреса. Деньги и вердикт не затрагиваются.

Сервер не может: изменить текст, изменить хеш, не донести голос, повлиять на вердикт, сдвинуть деньги, отрицать факт принятия задания, помешать донору или стримеру опубликовать текст самостоятельно.

Сервер может: не показать текст в своём UI. Хеш остаётся, процесс продолжается.

## 10. Обновление игры

Канистра blackholed. Обновление — новая канистра; threshold-ключи дериватны от `canister_id`, у новой они другие сами по себе.

- UI переключается на новую канистру.
- Старая перестаёт принимать задания, доживает открытые.
- Эскроу старой версии ссылаются на её ключ и разрешаются ею же.
- Заблэкхоленная канистра не удаляется. При исчерпании циклов она замерзает, состояние сохраняется, пополнить может кто угодно.
- Эскроу, чей резолвер замолчал, выходит через `refund()` по дедлайну.

Блэкхол — прод-шаг, зеркало S5 ядра и F5 фабрики: до выхода в прод канистра передеплоивается свободно.

## 11. Принятые ограничения

Не латать, а принять.

1. **Вес голоса стоит стримеру 3% от номинала, любому другому — 100%.** Репутация начисляется за деньги, отданные стримеру, и стример может отдать их себе. Механизм этого не предотвращает. Предотвращает публичность голосования и репутационная цена разоблачения.
2. **Снапшота веса нет.** Вес докупается во время голосования. Донат за минуту до голоса не учтён: книга отстаёт на финальность плюс ингест.
3. **Голосование чейн-локально.** Репутация на другой сети права голоса не даёт.
4. **Голос — ingress-вызов, оплачивается канистрой.** Спам подписями жжёт циклы. Митигация — rate limit на boundary nodes и бюджет, не код.
5. **Возврат денег не гарантируется.** Любой донат считается ушедшим навсегда. UI не обещает возврата ни при каком исходе.
6. **Заработать больше внесённого нельзя.** Ни донор, ни стример, ни голосующий. Приза не существует.
7. **Репутация начисляется только при `settle`.** Отказ, таймаут и рефанд репутации не дают.
8. **`duration` задаёт донор.** Заведомо короткий срок означает, что стример не примет задание и деньги вернутся.
9. **Конфиденциальность текста до принятия обеспечивается сервером, не криптографией.** Сервер держит plaintext, потому что модерирует его на входе. Преждевременное раскрытие возможно и механизмом не предотвращается. Подделка текста невозможна: хеш в канистре.
10. **Вердикт — подпись, исполнение — транзакция.** `settle`, который никто не исполнил до `DEADLINE`, проигрывает `refund()`. Принято: `DEADLINE_MARGIN` даёт 72 часа, `claim` доступен любому с момента вердикта.
